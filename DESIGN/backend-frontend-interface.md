# API Interface (gRPC / Connect-RPC)

本プロジェクトでは、Google発祥の技術である **Protocol Buffers (Protobuf)** と **Connect-RPC** を全面的に採用する。
共通のスキーマ定義 (`diary.proto`) を基盤とし、バックエンド (Go) とフロントエンド (Next.js) の間で型安全な通信を行う。

## 方針

- **統合管理**: 全ての通信（参照、更新、ストリーミング、エクスポート）を単一のサービスインターフェースに集約する。
- **逐次応答**: AIの回答生成はサーバーサイドからのストリーミング配信を行い、低遅延なUI体験を提供する。
- **セキュア通信**: 認証情報および API キーは Cookie (`HttpOnly`, `Secure`) を通じて自動的に付与される。
- **データ構造の正準 (SSOT)**: 各メッセージ（エンティティ）の具体的なフィールド定義については、 **`dataschema.md` を唯一の正解とする**。本ドキュメントでは RPC メソッドのインターフェース定義に専念する。

## サービス定義: DiaryService

### 1. 日記操作 (Diary Operations)

| メソッド名 | 役割 | 入力パラメータ | 返却値 |
| :--- | :--- | :--- | :--- |
| `ListDiaries` | 日記一覧の取得 | 期間フィルタ (`after`, `before`), 取得件数, カーソル | 日記リスト, 次ページカーソル |
| `GetDiary` | 単一日記の取得 | 日記ID | 日記エントリ (`DiaryEntry`) |
| `CreateDiary` | 日記の新規作成 | (将来拡張用パラメータ) | 作成された日記エントリ |
| `UpdateDiary` | 日記の部分更新 | 日記ID, トピック(任意), 本文(任意) | 更新後の日記エントリ |
| `DeleteDiary` | 日記の削除 | 日記ID | (空) |

### 2. AI対話 (AI Chat Streaming)

AIとの対話はストリーミング形式で実行される。

| メソッド名 | 役割 | 入力パラメータ | 返却値 (Stream) |
| :--- | :--- | :--- | :--- |
| `Chat` | AIとの対話 | 日記ID, ユーザー入力メッセージ | **ChatResponse (逐次送信)** |
| `UpdateChatFeedback` | フィードバック更新 | 日記ID, チャットID, 評価値 (-1, 0, 1) | 更新後のチャットエントリ |

**ChatResponse の挙動:**
1.  **トークン送信**: AIが回答を生成するたびに、細切れの文字列（トークン）を送信する。
2.  **完了通知**: 回答が完了した際、最終的なチャット履歴オブジェクト (`ChatEntry`) を一括送信する。
3.  **エラー通知**: 途中でエラーが発生した場合は、エラーメッセージを送信して接続を終了する。

#### 2.1 ストリーミングの制御 (Control Flow)

AIの回答生成中における中断および再送に関して、**ネットワーク層のキャンセルと、アプリケーション層の論理的な中断（Logical Interruption）を併用**することで、確実な制御を実現する。

- **中断 (Interrupt)**:
    - ユーザーが割り込んで新たな発言を行う際、フロントエンドは現在のストリーム（RPCコール）をキャンセルし、即座に新しい `Chat` リクエストを送信する。
    - バックエンドは以下の2つの機序により、古い生成プロセスを停止する：
        1.  **Context Cancel (Network)**: クライアント切断を検知し、リソースを解放する（基本動作）。
        2.  **Logical Check (Application)**: 新しい `Chat` リクエストにより、日記 (`DiaryEntry`) の「最新発言ID (`latestChatId`)」が更新される。古い生成プロセスは、次回トークン生成時にこの不一致を検知し、自ら処理を終了する（Self-Termination）。これにより、ネットワーク切断の検知漏れや遅延があった場合でも、確実に古い生成を破棄できる。
- **再開・リトライ (Resume/Retry)**:
    - 通信エラー等でストリームが予期せず切断された場合、バックエンド側でコミット（履歴への保存）が完了していなければ、フロントエンドは同一内容のリクエストを再試行することができる。
    - ただし、AIの生成は冪等ではないため、再試行時は最初からの生成となる。
- **状態同期**:
    - ストリームの最後で送信される `complete_entry` (ChatEntry) がクライアントに到達し、正常に受理された時点で、その対話セッションの履歴がフロントエンド・バックエンド間で同期されたものとみなす。

### 3. 要約操作 (Summary Operations)

| メソッド名 | 役割 | 入力パラメータ | 返却値 |
| :--- | :--- | :--- | :--- |
| `ListSummaries` | 要約一覧の取得 | 粒度 (週/月), 期間フィルタ, 件数, カーソル | 要約リスト, 次ページカーソル |
| `GetSummary` | 単一要約の取得 | 要約ID | 要約データ (`Summary`) |
| `CreateSummary` | 要約の新規作成 | 対象基準日時, 粒度 | 作成された要約データ |
| `UpdateSummary` | 要約の更新 | 要約ID, トピック(任意), 要約本文(任意) | 更新後の要約データ |
| `DeleteSummary` | 要約の削除 | 要約ID | (空) |

### 4. エクスポート操作 (Export Operations)

日記および要約データを外部に出力する。セルフホストモードにおける保存形式との互換性を重視する。

| メソッド名 | 役割 | 入力パラメータ | 返却値 |
| :--- | :--- | :--- | :--- |
| `ExportDiary` | 単一日記出力 | 日記ID, フォーマット ("md") | `ExportResponse` |
| `CreateMigrationPackage` | マイグレーションパッケージ作成 | 期間フィルタ, フォーマット | `MigrationPackageResponse` |

#### 4.1 エクスポート仕様

- **出力形式の厳格性**:
    - `ExportDiary` が返却する Markdown 文字列は、`dataschema.md` で定義された **「Markdown ストレージ・エクスポート形式」に厳格に準拠** しなければならない。
- **マイグレーション・パッケージとしての実装**:
    - `CreateMigrationPackage` は、単なる日記のアーカイブではなく、**セルフホストモードのデータディレクトリ (`storage_root`) をそのまま再現した ZIP アーカイブ** を生成する。

    - **同梱されるもの**: 
        - 全期間の日記 (`diaries/`)
        - 全期間の要約 (`summaries/`)
        - ユーザープロファイル・ペルソナ・コンテキスト (`user/profile.json`)
        - システム設定 (`user/settings.json`)
        - メタデータ (`export_metadata.json`)
    - これにより、ユーザーはクラウド版からダウンロードした ZIP をローカルの指定ディレクトリに展開するだけで、設定や会話履歴を含めたすべての環境移行を完了できる。

### 5. 認証・設定 (Auth & Settings)

| メソッド名 | 役割 | 入力パラメータ | 返却値 |
| :--- | :--- | :--- | :--- |
| `LoginWithGoogle` | Googleログイン | Google ID Token | ユーザーID, メールアドレス |
| `Logout` | ログアウト | (なし) | (空) |
| `UpdateUserProfile` | ユーザープロファイル更新 | ニックネーム(任意), 地域設定(任意) | 更新後のプロファイル |
| `SetGeminiApiKey` | APIキー設定 | Gemini API キー文字列 | (空) |

#### 5.1 APIキーの保持とセキュリティ (BYOK)

`SetGeminiApiKey` RPC 呼び出し時の挙動は、運用モードにより異なる。

- **クラウドモード**: 
    - バックエンドは受け取った API キーを DB に保存せず、レスポンスの **`Set-Cookie` ヘッダー**（`HttpOnly`, `Secure`, `SameSite=Strict`）を用いてブラウザに保持させる。
    - 以降の AI 連携を伴う RPC リクエストでは、ブラウザがこの Cookie を自動送信し、バックエンドはメモリ上でのみキーを展開して使用する。
    - **エクスポート時の制限**: クラウドモードでは API キーはサーバー側に保存されないため、 `ExportDiary` や `CreateMigrationPackage` によるエクスポートには含まれない。セルフホストへの移行後は、ユーザーが再度 API キーを設定する必要がある。
- **セルフホストモード**:
    - バックエンドは受け取った API キーをローカルの設定ファイル（`settings.json`）に永続化する。

## 主要なデータ構造

詳細なフィールド定義については、 **`dataschema.md` を唯一の正解（SSOT）とする。**

- **DiaryEntry**: 日記の本体。ID、トピック、本文、作成/更新日時、天気、ニュース、チャット履歴を含む。
- **ChatEntry**: 個別の発言内容。ID、発言者、内容、時刻、フィードバックを含む。
- **Summary**: 期間要約。ID、粒度、対象期間、対象日記IDリスト、要約本文を含む。
- **UserProfile**: ユーザー個別のペルソナ、会話コンテキストなどの管理。
- **DiaryListItem**: 一覧表示用の軽量なデータ。
- **ExportResponse**: Markdown形式のエクスポート結果。
- **MigrationPackageResponse**: 一括バックアップ/移行用パッケージ。

## 共通仕様

### 期間フィルタの扱い

`after`（開始）と `before`（終了）の指定による挙動は以下の通りとする。

- いずれも指定日時を **含む (inclusive)** 判定を行う。
- `after > before` のような不正な期間指定が行われた場合、サーバーは `InvalidArgument` エラーを返却する。
- 未指定の場合は「全期間」として扱う。

### エラーハンドリング

Connect-RPC の標準エラー体系に基づき、以下のステータスコードを返却する。

- `InvalidArgument`: リクエストパラメータの不備。
- `Unauthenticated`: 未ログイン状態でのアクセス。
- `PermissionDenied`: 操作権限がない（他人のデータへのアクセス等）。
- `NotFound`: 要求されたリソースが存在しない。
- `AlreadyExists`: 既に存在するリソースを作成しようとした場合。
- `ResourceExhausted`: 割り当て制限の超過（デモモードの利用制限等）。
- `Internal`: サーバー内部での予期せぬエラー。
